#+TITLE: Spec: Arrow-Based Input for High-Level MarkerTable API (from-Arrow Paths)
#+AUTHOR: rsx-rs developers
#+DATE: 2026
#+OPTIONS: toc:2 num:nil

* Status
v0.1 implementation merged.  See =rsx-core/src/source/= and the
=*_from_arrow= entry points in =rsx-python/src/lib.rs=.

The implemented design closely follows this spec:
- =MarkerStream= trait + blanket impl for =MarkersTableStream=, plus
  =ArrowMarkerSource= (in-memory) and =ParquetMarkerSource= (spilled).
- Decision wrapper =MarkerTableSource::from_arrow_ipc= picks in-memory
  or Parquet spill using a Beissinger-style working-set estimate.
- =RSX_FORCE_SPILL= and =RSX_SPILL_BYTES= env vars override the
  heuristic (tests use the first; debugging uses the second).

What this spec calls =MarkerTableSource= maps to two pieces in code:
- =rsx_core::source::MarkerStream= (the trait every command consumes), and
- =rsx_core::source::MarkerTableSource= (the enum the bindings build via
  =from_arrow_ipc=, which is what implements =MarkerStream= for the
  from-Arrow code path).

* Goals
- Provide a clean, zero-visible-temp-file experience for users who pass DataFrames to the high-level =MarkerTable= API (= .freq, .depth, .distrib, .signif, etc.).
- Support both fully in-memory Arrow processing (fast path for moderate data) and efficient spill-to-disk for large data.
- Use Parquet as the only spill format when disk is required.
- Keep the design DRY: a single shared helper that all simple table-oriented commands can use.
- Respect the core low-memory streaming design of rsx-rs; the new paths are additive and only used by the DataFrame entry points.
- Base spill decisions on literature-derived size prediction rather than magic constants.

* Non-Goals (for this iteration)
- Replacing or modifying the existing streaming =MarkersTableStream= / file-based paths.
- Making the helper the single engine for the entire CLI or low-level API.
- Full out-of-core algorithms for every command (we may still stream from the spilled Parquet, but we do not promise perfect tiling for every statistic yet).

* Scope
This spec covers only the =*_from_arrow= entry points called from the high-level Python API when a user does:

#+begin_src python
mt = MarkerTable.from_dataframe(df)
mt.freq(...)
mt.depth(...)
mt.distrib(...)
mt.signif(...)
#+end_src

The classic path-based usage (=MarkerTable.from_path(...)= or direct low-level calls) continues to use the existing streaming code and is out of scope for the spill logic described here.

* Data Flow (DF Path)
1. Python side converts the user's DataFrame (via narwhals) to a =pyarrow.Table=.
2. The table is serialized to Arrow IPC bytes in RAM (no Python temp file).
3. Bytes are passed across the FFI boundary to a new =*_from_arrow= function.
4. On the Rust side the bytes are turned into =RecordBatch=es.
5. A shared helper (=MarkerTableSource=) decides whether to keep the data in memory or spill to a temporary Parquet file.
6. The command logic runs against the in-memory view or the spilled Parquet.
7. Results are returned as Arrow (via the existing pure-IPC machinery) or as a =TableResult=.
8. Any internal Parquet spill is deleted before the function returns to Python.

* Proposed Abstraction: MarkerTableSource

** Responsibilities
- Own the source of truth for a marker depth table that arrived via Arrow.
- Provide uniform access to:
  - Number of individuals and their names
  - Per-marker depth vectors (after min-depth filtering)
  - Group masks when a popmap is also supplied as Arrow
- Hide whether the underlying storage is in-memory Arrow batches or a spilled Parquet file.
- Expose a size estimator and a spill decision.

** Sketch (Rust)

#+begin_src rust
pub enum MarkerTableSource {
    InMemory { batches: Vec<RecordBatch>, min_depth: u16, ... },
    SpilledParquet { path: PathBuf, min_depth: u16, ... },
}

impl MarkerTableSource {
    pub fn from_arrow_ipc(
        bytes: &[u8],
        min_depth: u16,
        popmap_bytes: Option<&[u8]>,
        estimator: &SizeEstimator,
    ) -> Result<Self> { ... }

    pub fn n_individuals(&self) -> usize { ... }
    pub fn individual_names(&self) -> &[String] { ... }

    /// Iterate markers with filtered depths (works for both variants)
    pub fn for_each_marker<F>(&self, f: F) -> Result<()>
    where F: FnMut(u32, &str, &[u16]);

    pub fn estimated_working_set_bytes(&self) -> u64 { ... }
    pub fn is_spilled(&self) -> bool { ... }
}
#+end_src

The exact method names and internal representation are open for review.

* Size Prediction / Spill Decision (Literature-Backed)

** Core Idea
We want a defensible heuristic that answers:

"Given what we know about this dataset, will the working set required by this command comfortably fit in available RAM?"

If the answer is "no", we spill the incoming Arrow data to a temporary Parquet file before proceeding.

** First-Cut Estimator (v0.1) – Beissinger-based

We combine two sources:

1. Beissinger et al. (2013) – "Marker Density and Read Depth for Genotyping Populations Using Genotyping-by-Sequencing" (*Genetics*).

   Provides statistical models to predict the expected number of usable markers *M* as a function of:
   - Number of samples *N*
   - Target sequencing depth / coverage
   - Restriction enzyme (6-cutter vs 8-cutter, etc.)
   - Genome size (or effective fraction cut)
   - Downstream filtering thresholds (minimum depth, missing data rate, etc.)

   A simplified form that is practical to implement:

   #+begin_example
   M_pred ≈ f(N, depth, genome_size, enzyme, min_depth, max_missing)
   #+end_example

   In code we can start with a table-driven approximation or a lightweight regression fitted on published GBS/RAD-seq datasets, then allow users to override with known experimental parameters.

2. Empirical overhead factors drawn from the literature (TASSEL-GBS, Stacks, Torkamaneh et al. 2016, ipyrad, etc.):

   - Raw dense matrix size:  *N* × *M* × bytes_per_cell
     - u16 depth → 2 bytes
     - Two u16 (ref+alt) → 4 bytes
   - Typical in-memory multipliers when using high-level languages + temporary structures:
     - Python pandas / R data.frame: 4–8×
     - Arrow in-memory (with validity buffers, etc.): 1.5–3×
     - Our command working set (bitsets, group masks, per-marker accumulators): additional 1.5–4×
   - Conservative default overhead factor we will use initially: **6×**

** Concrete Formula (first implementation)

#+begin_src rust
fn estimate_working_set_bytes(
    n_samples: usize,
    m_observed_or_predicted: usize,
    bytes_per_cell: u64,           // 2 or 4
    overhead_factor: f64,          // default 6.0
    command_specific_multiplier: f64,  // e.g. 1.3 for freq, 2.0 for distrib/signif
) -> u64 {
    let raw = (n_samples as u64) * (m_observed_or_predicted as u64) * bytes_per_cell;
    (raw as f64 * overhead_factor * command_specific_multiplier) as u64
}
#+end_src

When the data arrives as Arrow batches we also have an *observed* *M*.  We take the maximum of the observed *M* and the Beissinger-predicted *M* (if parameters were supplied) to be conservative.

** Spill Threshold

We compare the estimated working set against a configurable fraction of total system RAM (queried via =sysinfo= or similar) with headroom.

Example policy (tunable via environment variable or API):

- If estimated working set > 55 % of available RAM → spill to Parquet.
- User may force spill or force in-memory via a parameter.

The Parquet file is written with Arrow's Parquet writer (columnar, snappy/zstd compression) and is deleted on drop / function exit.

** Future Refinements (not for v0.1)
- Use a more precise Beissinger model or a small regression trained on our own benchmark collection.
- Account for de novo vs. reference-based locus discovery (the former has higher temporary memory during catalog construction).
- Add a "dry-run" mode that only reports the estimated size without doing any work.

* Impact on Existing Benchmarks

- All new code lives behind the existing =arrow-output= (or a new =arrow-input=) feature flag.
- The classic streaming benchmarks continue to exercise only the old paths.
- We will add a new micro-benchmark category ("arrow-ingestion") that measures the from-Arrow path with both in-memory and spilled-Parquet variants.  These will not be part of the "must not regress" gate for the streaming code.

* Open Questions

- Exact name of the helper type.
- Whether popmap is passed as a second set of IPC bytes or as a separate struct.
- How much of the Beissinger model we implement in v0.1 vs. stub + documentation.
- Whether we expose the spill decision / estimated size to the Python caller (useful for diagnostics).
