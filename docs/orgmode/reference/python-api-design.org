* Python API Design

The current Python bindings in =rsx-python= provide thin, direct wrappers around the Rust command-line functions via PyO3. While functional, they are low-level and not idiomatic for Python users in the scientific computing and bioinformatics ecosystem.

This document proposes a high-level, object-oriented Python API layer that follows Python best practices, provides excellent DataFrame interoperability, and enables compelling end-to-end workflows (including plotting) within a single Python script or notebook.

**Status**: Design phase. Implementation will live in pure Python on top of the existing low-level bindings (kept private as =pyrsx._rsx=).

* Goals

- **Idiomatic Python**: Classes, methods, rich return values, proper =__repr__=, context managers where appropriate, and excellent editor support via type stubs.
- **DataFrame Agnostic**: Work seamlessly with =pandas=, =polars=, =pyarrow=, and future DataFrames via a lightweight adapter layer (using =narwhals=).
- **Progressive Disclosure**: Low-level power remains available; the high-level API is the default pleasant experience.
- **Rich Results + Visualization**: Result objects carry DataFrames + metadata and provide =plot_*()= methods for immediate insight.
- **Reproducibility & Provenance**: Explicit parameter objects, summary reports, and clear mapping back to the underlying Rust calls and proofs.
- **Paper-Ready Demo**: Enable a single, beautiful Python script that demonstrates rsx + analysis + plotting for the BMC paper and documentation.

* Proposed Architecture

**Layering**

- =pyrsx._rsx= :: Thin PyO3 bindings (file/bytes in, file/bytes or raw structs out). Stable, not for direct user consumption.
- =pyrsx.api= :: High-level pure-Python classes and functions (the public API).
- =pyrsx._adapters= :: DataFrame compatibility layer (narwhals-based).
- =pyrsx.plotting= :: Matplotlib/seaborn (default) + optional Plotly backend.

**Core Objects**

#+begin_src python
from pyrsx import MarkerTable, TriageResult, PcaResult

# Construction from multiple sources
table = MarkerTable.from_path("markers.tsv")
table = MarkerTable.from_dataframe(df, backend="auto")  # pandas, polars, etc.

# Fluent API returning rich objects
triage: TriageResult = table.triage(
    popmap=popmap_path_or_df,
    min_depth=10,
    posterior_threshold=0.9,
    prior=0.01,
    linked_prob=0.9,
)

# Introspection and export
print(triage.summary())
df = triage.to_dataframe()          # same flavor as input, or pandas default
triage.to_pandas()
triage.to_polars()

# Built-in, publication-quality plotting
triage.plot_evidence()
triage.plot_posterior_vs_bayes()
triage.plot_pca(hue="candidate_class")

# Parameter objects for complex configuration (dataclasses)
from pyrsx.api.params import TriageParams
params = TriageParams(min_depth=5, output_bayes=True, ...)
result = table.triage(params=params)
#+end_src

**DataFrame Adapters**

We use =narwhals= (lightweight, zero heavy dependencies) to implement a =DataFrameLike= protocol.

- Input: Any object that =narwhals= can wrap is accepted.
- Output: Methods like =to_dataframe(backend="pandas")= or automatic "echo the input flavor".
- This follows the pattern used by modern libraries (e.g., parts of the Polars/Pandas ecosystem in 2025+).

This avoids hard dependencies on pandas while giving users exactly what they expect.

* Detailed Class Sketch

**MarkerTable**

Central handle to a marker depth table.

- Construction: =from_path=, =from_dataframe=, =from_parquet= (future).
- Methods: =triage=, =pca=, =depth_stats=, =distrib=, =freq=, =map= (alignment), etc.
- Properties: =n_markers=, =n_individuals=, =schema=.
- =__repr__= and =summary()= showing shape, depth distribution, etc.

**TriageResult**

Returned by =MarkerTable.triage(...)=.

Contains:
- The per-marker results (strict call, posterior, BF, penetrance, bias, candidate_class).
- The input parameters (for reproducibility).
- Methods: =to_dataframe()=, =summary()=, =plot_evidence()=, =plot_posterior_distribution()=, =filter_by_class()=.

Similar rich objects for =PcaResult=, =DepthStats=, =DistributionResult=.

**Parameter Objects**

Use =dataclasses= (or =pydantic= if we decide on validation) for complex option sets:

#+begin_src python
@dataclass
class TriageParams:
    min_depth: int = 1
    signif_threshold: float = 0.05
    posterior_threshold: float = 0.9
    bayes_factor_threshold: float = 10.0
    prior: float = 0.01
    linked_prob: float = 0.9
    correction: str = "bonferroni"
    test: str = "chisq"
    # ...
#+end_src

This makes the API self-documenting and easy to serialize for reproducibility.

* Plotting Integration

Result objects will grow =plot_*= methods using =matplotlib= + =seaborn= by default (the standard for papers).

- =TriageResult.plot_evidence()= : stacked bar or grouped bars showing strict / posterior-only / BF-only partitions per dataset or overall.
- =TriageResult.plot_posterior_vs_bayes()= : scatter or hexbin to show the relationship that justifies the hybrid rule.
- =PcaResult.plot()= : scatter of PC1 vs PC2, colored by phenotypic sex and/or posterior probability, with outlier annotation.

Backends can be swapped (Plotly for notebooks, etc.) via a small registry.

This directly supports the "single script that produces insight + figures" story for the BMC paper.

* Implementation Roadmap (Option B — more complete from day one)

1. **Adapter Layer** — Narwhals wrapper + conversion utilities. Tests against pandas + polars.
2. **Core Objects** — =MarkerTable=, =TriageResult=, =PcaResult= (with full =__repr__=, =summary()=, DataFrame export).
3. **Parameter Dataclasses** — =TriageParams=, =PcaParams=, etc.
4. **Plotting** — At least the two most important plots for the biology narrative (evidence partition + PCA).
5. **High-level Entry Points** — Functional wrappers that return the rich objects (for users who prefer functions).
6. **Documentation & Example** — This design doc + a polished =examples/end_to_end_demo.py= that runs on the literature data and produces figures we can use in the paper.
7. **Type Stubs & Editor Experience** — Update =pyrsx.pyi= and ensure =__all__=, docstrings, etc. are excellent.
8. **Error Handling** — Map Rust errors to appropriate Python exceptions with context.
9. **Packaging** — Keep low-level in the Rust extension; high-level pure Python in the wheel.

* Example End-to-End Script (Paper-Ready)

#+begin_src python
#!/usr/bin/env python3
"""End-to-end demonstration of the high-level pyrsx API.

Runs the full triage workflow on a literature-derived marker table
and produces publication-quality figures in a single script.
"""

import pyrsx as rsx
import pandas as pd
import seaborn as sns
import matplotlib.pyplot as plt

# 1. Load (or generate) a marker table — DataFrame native
markers = pd.read_csv("markers.tsv", sep="\t", comment="#")

# 2. Construct the central object (accepts pandas here)
table = rsx.MarkerTable.from_dataframe(markers)

# 3. Run the biologically interesting triage (hybrid strict + posterior)
triage = table.triage(
    popmap="popmap.tsv",
    min_depth=10,
    posterior_threshold=0.9,
    prior=0.01,
)

# 4. Immediate insight
print(triage.summary())

# 5. Beautiful figures with almost no code
fig, ax = plt.subplots(figsize=(8, 5))
triage.plot_evidence(ax=ax)          # shows strict / posterior-only / BF-only
plt.title("Evidence classes recovered by rsx triage (depth ≥ 10)")
plt.tight_layout()
plt.savefig("triage_evidence_classes.png", dpi=200)

triage.plot_pca(hue="posterior_sex_linked")
plt.savefig("triage_pca.png", dpi=200)

# 6. Export for further analysis in any DataFrame library
df = triage.to_polars()   # or .to_pandas() — matches user preference
df.write_parquet("triage_results.parquet")
#+end_src

This script (or a version using the real literature data from the benchmark harness) is exactly what we can show in the "Binding-backed workflow evidence" section of the BMC paper and in the repository README.

* Open Questions & Decisions

- Dependency on =narwhals= (very small, recommended) vs. pure duck-typing.
- Whether to use =pydantic= for parameter validation (nice but adds a dep).
- Default plotting backend (matplotlib is safest for papers; we can make it pluggable).
- How aggressively to deprecate the old low-level functions (keep them for at least one release).
- Whether =MarkerTable= should be mutable or immutable (immutable + =with_*= methods is more functional/Pythonic in many cases).

* References & Prior Art

- Existing low-level bindings in =rsx-python/src/lib.rs= and =python/pyrsx/=.
- =narwhals= documentation for DataFrame interoperability.
- Design of similar high-level APIs in =pysam=, =scanpy=, =polars= lazy API, and =pybedtools=.
- The three algorithm pseudocode blocks and numerical proofs already present in the main paper (which this API will expose cleanly).

Once approved, this design will guide the implementation of the high-level Python surface that makes rsx feel like a first-class citizen in the Python bioinformatics world.
